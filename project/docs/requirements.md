# 语音转文字事务记录网页 需求基线 V1.1
> 最后更新：2026-06-24
> 本文档由需求管家 V5.0 维护

## 一、需求清单

### 1.1 认证与用户管理

#### REQ-001 本地账号登录与会话管理
**需求陈述**：系统支持用户通过账号密码登录，登录成功后进入事务记录工作台，并保留当前登录态。

**验收标准**：
1. Given 用户输入正确账号和密码，When 点击登录，Then 系统跳转到工作台并显示当前用户信息。
2. Given 用户未登录，When 访问受保护页面，Then 系统跳转到登录页。
3. Given 用户主动退出，When 退出成功，Then 系统清除本地会话并返回登录页。

**异常处理**：
- 账号或密码错误：提示登录失败原因，不泄露敏感信息。
- 会话失效：提示重新登录并跳转登录页。

#### REQ-002 微信网页扫码登录
**需求陈述**：系统支持微信网页扫码登录，用户可通过微信开放平台 OAuth 流程完成身份认证并进入系统。

**验收标准**：
1. Given 用户位于登录页，When 点击微信登录，Then 系统跳转到后端生成的微信授权入口。
2. Given 微信授权成功且能够识别用户身份，When 回调完成，Then 系统创建或绑定本地用户并进入工作台。
3. Given 系统未配置微信 AppID 或 AppSecret，When 用户点击微信登录，Then 系统给出明确提示或走模拟登录流程，不直接报错崩溃。

**异常处理**：
- 微信授权失败：提示授权失败原因并允许重新登录。
- 微信回调缺少必要参数：拒绝登录并记录服务端日志。

#### REQ-013 第三方 OAuth2.0 账号绑定（GitHub 优先）
**需求陈述**：系统支持在“账号管理”中绑定第三方 OAuth2.0 身份提供方账号，优先实现 GitHub 账号绑定，并为未来扩展更多提供方预留统一模型。

**验收标准**：
1. Given 用户已登录，When 进入账号管理并点击“绑定 GitHub”，Then 系统跳转到 GitHub 授权页面并完成回调。
2. Given 授权成功，When 回到系统，Then 当前账号展示“GitHub 已绑定”，且不会新建重复用户。
3. Given 该 GitHub 账号已绑定其他本地账号，When 尝试绑定，Then 系统提示绑定冲突并拒绝操作。
4. Given 用户解除绑定，When 解绑成功，Then 账号绑定状态立即更新，并保留至少一种可用登录方式。

**异常处理**：
- 未配置 GitHub ClientId/Secret：提供模拟绑定流程或明确提示，不直接崩溃。
- 回调缺少 code/state 或 state 校验失败：拒绝绑定并提示原因。

#### REQ-003 用户增删改查
**需求陈述**：管理员可创建、编辑、禁用和删除用户，并为用户分配一个或多个角色。

**验收标准**：
1. Given 当前用户具备用户管理权限，When 新增用户，Then 系统保存账号、姓名、邮箱、角色和状态信息。
2. Given 当前用户具备用户管理权限，When 编辑用户，Then 系统更新用户资料并立即反映在列表中。
3. Given 当前用户具备用户管理权限，When 禁用用户，Then 被禁用用户无法继续登录。
4. Given 当前用户具备用户管理权限，When 删除用户，Then 系统执行逻辑删除并从默认列表中隐藏。

**异常处理**：
- 账号重复：阻止保存并提示修改账号。
- 删除当前登录用户：禁止操作并提示原因。

#### REQ-004 角色与权限管理
**需求陈述**：系统支持管理员维护自定义角色，并为角色配置页面访问和操作权限。

**验收标准**：
1. Given 当前用户具备角色管理权限，When 新增角色，Then 系统保存角色名称、描述和权限集合。
2. Given 角色已分配给用户，When 修改角色权限，Then 相关用户在下一次鉴权时按最新权限生效。
3. Given 当前用户不具备某页面访问权限，When 访问该页面，Then 系统拒绝访问并提示无权限。

**异常处理**：
- 角色名称重复：阻止保存并提示修改名称。
- 删除系统默认角色：默认角色不可物理删除，仅允许禁用或调整权限。

#### REQ-005 企业单点登录预留
**需求陈述**：系统预留企业单点登录接入能力，当前版本至少提供统一配置入口和扩展点，且不影响本地登录与微信登录并存。

**验收标准**：
1. Given 管理员进入系统配置，When 查看认证设置，Then 可见单点登录相关配置项或占位说明。
2. Given 后续接入外部身份提供方，When 扩展认证模块，Then 不需要重写现有用户和角色模型。

**异常处理**：
- 未配置单点登录：系统默认使用本地账号密码登录。

### 1.2 事务记录

#### REQ-006 待办事务记录管理
**需求陈述**：用户可创建、查看、编辑、删除和检索待办事务记录，每条记录至少包含标题、正文、状态、优先级、开始时间、结束时间、创建时间和更新时间。

**验收标准**：
1. Given 用户已登录，When 新增事务记录，Then 系统保存记录并显示在列表中。
2. Given 事务记录已存在，When 编辑内容或状态，Then 系统更新记录并刷新更新时间。
3. Given 用户输入关键词或状态筛选条件，When 执行搜索，Then 列表只展示符合条件的记录。
4. Given 普通用户查看列表，When 未拥有跨用户查看权限，Then 只能看到自己创建的记录。

**异常处理**：
- 必填项缺失：阻止保存并提示缺失字段。
- 记录不存在：提示记录已失效并刷新列表。

#### REQ-007 事务状态流转
**需求陈述**：系统至少支持“待处理、进行中、已完成”三种事务状态，并允许用户快速切换。

**验收标准**：
1. Given 用户打开事务详情或列表，When 修改状态，Then 新状态立即保存并反馈成功。
2. Given 用户按状态筛选，When 选择某一状态，Then 列表仅显示对应记录。

**异常处理**：
- 无权限修改他人记录：禁止保存并提示原因。

#### REQ-008 日历视图查阅
**需求陈述**：系统提供日历界面管理事务，支持按月、按周、按 3 日、按日四种视图查阅事务安排，并可从日历中查看或编辑事务详情。

**验收标准**：
1. Given 用户进入日历页，When 切换到月视图，Then 可按月查看所有可见事务的日期分布。
2. Given 用户进入日历页，When 切换到周视图、3 日视图或日视图，Then 系统按对应时间范围展示事务。
3. Given 某条事务设置了开始时间和结束时间，When 在日历中展示，Then 系统按时间位置或日期归属正确显示。
4. Given 用户点击日历中的事务，When 打开详情，Then 可查看完整内容并根据权限进行编辑。

**异常处理**：
- 事务未设置开始时间：在列表中提示未排期，并不出现在需要时间轴定位的视图块中。
- 时间范围跨天：在日历中按跨天事务处理，不丢失显示。

### 1.3 语音转文字

#### REQ-009 funASR 语音转文字接入
**需求陈述**：系统支持用户上传音频文件，由 Express TypeScript 后端调用 funASR 服务完成语音转文字，并将识别结果回填到事务记录正文或语音摘要区域。

**验收标准**：
1. Given 用户在新建或编辑事务时上传音频，When funASR 返回识别结果，Then 系统在页面展示转写文本并允许继续编辑。
2. Given 用户确认使用转写结果，When 保存事务，Then 转写后的文本被写入事务记录。
3. Given 管理员已配置 funASR 服务地址，When 后端发起识别请求，Then 系统使用配置地址调用识别接口。

**异常处理**：
- 音频格式不支持：提示支持的格式并阻止提交。
- funASR 服务不可用或超时：提示识别失败，可稍后重试或手动输入。
- 识别结果为空：保留原内容并提示用户检查音频质量。

### 1.4 技术与数据要求

#### REQ-010 技术栈约束
**需求陈述**：前端使用 React + Ant Design，后端使用 Express + TypeScript，前后端分层实现并通过 HTTP API 通信。

**验收标准**：
1. Given 开发者查看前端项目，When 阅读依赖和页面代码，Then 可确认使用 React 与 Ant Design。
2. Given 开发者查看后端项目，When 阅读依赖和入口代码，Then 可确认使用 Express 与 TypeScript。
3. Given 前端调用业务接口，When 接口返回数据，Then 能通过统一 API 层完成页面渲染。

**异常处理**：
- 接口调用失败：前端提示失败原因并允许重试。

#### REQ-011 PostgreSQL 数据持久化
**需求陈述**：系统使用 PostgreSQL 存储用户、角色、用户角色关系、事务记录和系统配置数据，不再依赖浏览器本地存储作为主数据源。

**验收标准**：
1. Given 服务端启动，When 数据表不存在，Then 系统可自动初始化基础表结构和默认数据。
2. Given 用户或事务发生新增、修改、删除，When 操作完成，Then 数据被写入 PostgreSQL。
3. Given 服务重启，When 再次访问系统，Then 之前保存的数据仍可读取。

**异常处理**：
- 数据库连接失败：服务端输出明确日志，前端提示服务不可用。
- 默认数据库不存在或权限不足：启动时提示配置问题。

#### REQ-012 可维护性与配置化
**需求陈述**：系统采用前后端分层设计，关键能力通过环境变量配置驱动，便于后续扩展为真实生产系统。

**验收标准**：
1. Given 开发者查看项目结构，When 阅读说明文档，Then 能快速了解启动方式和关键配置项。
2. Given 需要切换 funASR 或认证配置，When 修改环境变量，Then 无需重写业务页面代码。

**异常处理**：
- 关键配置缺失：系统给出默认提示或降级行为，而不是直接崩溃。

## 二、术语表

| 术语 | 精确定义 |
|:---|:---|
| 事务记录 | 以待办事项为核心的业务记录，包含文本内容、状态和时间信息 |
| funASR | 外部语音识别服务，用于将上传音频转换为文本 |
| 角色 | 一组权限的集合，可分配给多个用户 |
| 权限 | 页面访问或具体操作能力，如查看用户列表、编辑角色、删除事务 |
| 微信网页扫码登录 | 通过微信开放平台的网页 OAuth / 扫码授权流程完成 Web 登录 |
| PostgreSQL | 本系统的主数据存储数据库 |

## 三、需求关系图谱摘要
- **硬依赖**：`REQ-008` 依赖 `REQ-006` 的事务时间字段。
- **硬依赖**：`REQ-009` 依赖 `REQ-006` 的事务编辑能力与 `REQ-010` 的后端 API。
- **硬依赖**：`REQ-003` 依赖 `REQ-004` 的角色模型。
- **软关联**：`REQ-005`、`REQ-002` 与 `REQ-001` 共用认证模块。
- **硬依赖**：`REQ-011` 为全部业务数据的底层支撑。
