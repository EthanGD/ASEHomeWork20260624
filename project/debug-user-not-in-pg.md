[OPEN] Debug Session: user-not-in-pg

## 症状
- 前端测试时新增了 `cys` 用户
- 在 PostgreSQL 中未查到该用户

## 期望
- 应用中的用户新增、修改、删除都应以 PostgreSQL 为主数据源
- 前端展示与 PostgreSQL 中的实际数据保持一致

## 假设（待证伪）
1. 前端某处仍在使用本地内存状态或浏览器存储，新增成功只是界面态，没有真正调用后端创建用户接口。
2. 前端调用了 `/api/users`，但因为代理、鉴权或请求体问题，请求失败或被拦截，界面误以为成功。
3. 后端 `/api/users` 命中了，但 SQL 插入失败或事务未提交，导致数据库中没有记录。
4. 用户其实已写入 PostgreSQL，但被连到了错误的数据库实例/端口，导致你查的是另一套库。
5. 新增的是微信用户或逻辑删除/筛选导致查不到，不是“未写入”而是“查错表/查错条件”。

## 采集计划
- 给前端 API 请求层与后端 `/api/users` 路由加最小埋点
- 复现“新增 cys 用户”
- 对比前端请求日志、后端接口日志和 PostgreSQL 查询结果

## 证据记录
- pre-fix:
  - `server/.env` 指向 `PGPORT=5432`
  - 查询结果：
    - 5432 / PG17 中 `cys_count = 0`
    - 5433 / PG18 中 `cys_count = 1`
  - 通过当前后端接口创建 `cys_check_1782359159015` 后：
    - 5432 中存在该用户
    - 5433 中不存在该用户
- post-fix:
  - 将后端配置切回 `PGPORT=5433`
  - 重新启动后端后，通过 `/api/users` 查询可见 `cys`

## 结论
- Root cause:
  - 应用确实在使用 PostgreSQL 管理业务数据，但后端曾被配置到另一套 PostgreSQL 实例（5432 / PG17）。
  - 你测试新增的 `cys` 在 5433 / PG18，而当前应用连接的是 5432，所以前端与数据库查询结果不一致。
- Fix:
  - 将 `server/.env` 中的 `PGPORT` 改回 `5433`
  - 重新启动后端后，`/api/users` 已能返回 `cys`
